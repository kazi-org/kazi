# ADR-0077: A CI-generated Claude Code plugin as an additional, lockstep-versioned distribution channel

## Status

Accepted

## Date

2026-07-18

## Refines / supersedes

Amends **ADR-0074** in one respect: `LOCAL.md`, the operator-owned extension
point, moves from *inside* the generated skill directory to a **stable path
outside any directory an installer or plugin update replaces wholesale**
(decision 3 below). Every other ADR-0074 decision stands unchanged -- the
self-contained multi-file skill, the three managed documents, and the
never-overwrite contract for `LOCAL.md` itself. ADR-0024's consent-first
principle is reaffirmed, not weakened: installing the plugin IS the opt-in.

## Context

Onboarding a Claude Code user to kazi takes three separate opt-in commands
(`kazi install-skill`, `kazi init --with-mcp` per repo, `kazi install-hooks`),
and the teaching surfaces those commands write only refresh when the operator
remembers to re-run them after a binary upgrade. That gap is a recurring,
OBSERVED staleness class, not a hypothetical (#1173):

- **#956** -- a tiering fix stranded in a local dotfile while the shipped
  guidance moved on.
- **#1166** (E56/T56.2) -- the embedded skill still teaching workarounds for
  flags the binary had since grown.

Claude Code plugins are the platform's native answer to exactly this class:
one install, marketplace-driven updates, and a manifest that bundles skills,
MCP server registrations, and hooks in a single artifact. The pattern is
established platform-side -- official plugins already bundle MCP servers with
skills and even manage server binaries -- so this is an integration with a
supported channel, not a novel mechanism.

The risk a second channel introduces is a second drift axis: a hand-maintained
plugin could lag the binary exactly the way a hand-run installer does today.
The design below exists to make that structurally impossible rather than
procedurally discouraged.

## Decision

1. **Ship a `kazi` Claude Code plugin, GENERATED BY CI from the same sources
   the existing installers use.** The plugin bundles: the rendered skill
   (`Kazi.Teach.InstallSkill`'s `skill_md/0`, `authoring_md/0`, `recipes_md/0`
   -- the ADR-0074 functions), the MCP server registration (the same
   `kazi mcp` stdio shape `init --with-mcp` writes, ADR-0044), and the bus
   hook declarations (T55.9/ADR-0076), declared in the plugin manifest instead
   of mutated into user settings. The generator is a RENDERING step over the
   existing single source of truth; if the manifest needs content the existing
   renderers do not produce, the renderer is extended -- the plugin generator
   never duplicates teaching logic.

2. **Lockstep versioning: the plugin version IS the release version.** The
   release pipeline regenerates the plugin from the just-built release
   artifact and publishes it to a `kazi-org` marketplace repo in the SAME CI
   run that publishes the binary -- never hand-bumped, never hand-published,
   so marketplace content can never lag the binary. Publishing is additive
   and failure-tolerant: a failed plugin-publish step must not block or fail
   the underlying binary release. Because the two channels (binary via
   brew/direct install, plugin via marketplace) can still be upgraded
   independently by the operator, a SessionStart check -- riding the existing
   T55.9/ADR-0076 hook machinery, not a new hook type -- compares
   `kazi version` against the installed plugin's declared version and emits a
   ONE-LINE warning on mismatch, silent on match, fail-silent on timeout.

3. **`LOCAL.md` moves to a stable path (the ADR-0074 amendment).** Plugin
   updates replace skill directories wholesale, so an operator extension file
   *inside* the skill directory would be destroyed on every marketplace
   update -- the exact drift-class ADR-0074 exists to prevent. The canonical
   `LOCAL.md` location becomes a stable path OUTSIDE any managed/replaced
   directory: `~/.claude/skills/kazi/LOCAL.md` remains the discovery path for
   explicit installs, and the shipped SKILL.md points at the stable location
   rather than a path relative to whatever directory the skill happens to be
   installed in. An existing `LOCAL.md` at the old location is detected and
   migrated or loudly warned about -- never silently ignored. The exact
   mechanics land in T61.2; this ADR fixes the contract: **no operator
   customization may live in a directory a plugin update replaces.** This
   benefits non-plugin installs equally (survives any future relocation of
   the skill directory).

4. **Non-goals.** The plugin does NOT replace or deprecate `install-skill`,
   `init --with-mcp`, or `install-hooks`. They remain the explicit-command
   channel for Codex/Cursor/other harnesses and for operators who prefer
   explicit commands (ADR-0024's consent precedent: explicit-first, both
   channels valid, neither "the" way). The plugin is an ADDITIONAL channel
   rendered from the same single source of truth -- never a fork of it.
   Nothing in this decision couples convergence or any core kazi behavior to
   the plugin's presence.

## Consequences

- One install (or one marketplace update) delivers skill + MCP + hooks
  together, and the teaching surfaces refresh with the binary release cadence
  instead of the operator's memory -- closing the #956/#1166 staleness class
  at the mechanism level for Claude Code users.
- The release pipeline gains a publish step and a marketplace repo to own;
  lockstep is enforced by construction (generated from the release artifact,
  in the release run), so there is no version field a human can forget to
  bump.
- Version skew between independently-upgraded channels is surfaced by a
  bounded one-line SessionStart warning, not silently absorbed (the honest
  companion to lockstep publishing).
- Operators with an existing in-directory `LOCAL.md` see a one-time
  migration/warning; thereafter their customization survives plugin updates,
  re-installs, and skill-directory relocations alike.
- Claude Code's plugin manifest schema is an external dependency; if it
  shifts before E61 ships, the generator targets the CURRENT schema at
  authoring time (T61.3 researches, never guesses) and a genuinely unstable
  schema is escalated to the maintainer rather than papered over.
- Existing-installer users see zero behavior change.
