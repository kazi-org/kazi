# E60 -- Fleet-wide visibility: cross-machine runs, ghost-row reaping, operator attention, portfolio view, cost report (P2)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

Five issues found via a full open-issue triage pass (2026-07-17), all
extending E46's fleet observability (UC-061 starmap, UC-062 replayable
trail) and UC-063's budget economics to the cross-machine reality E51/E55
already built for the bus: today the run registry and dashboard are still
per-machine only, so operating a fleet from one machine while another
machine runs apply loops shows nothing in flight. Two are data-hygiene bugs
in the existing single-machine run registry (T60.2's ghost rows). Three are
proposals riding the already-shipped bus/board machinery (T60.1's
cross-machine posting, T60.3's operator-attention board, T60.4's portfolio
view) that per ADR-0011's read-only-projection line must compose ONLY from
state kazi already objectively has -- no manual curation, no new
task-management surface. One is a small, self-contained terminal-report
addition (T60.5).

Acceptance: `kazi dashboard` on any machine shows in-flight runs from every
machine in the fleet (not just its own), via the bus; every run reaches a
terminal status eventually (no permanent ghost rows); a fleet-wide "waiting
on operator" view aggregates blocked-session pings from every machine; a
portfolio view (planned/in-progress/stuck/complete) is composed purely from
kazi's own objective state; `kazi apply`'s human-readable terminal output
shows a cost/token/iteration breakdown per goal.

Prefer (a) if a specific missed trigger is found; fall back to (b) as a safety net either way.
