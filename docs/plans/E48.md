# E48 -- Economy feedback loop: persisted run economics, learned budgets, behavior-first prompting, honest budget stops (P1, ADR-0058)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

An investigation of every `over_budget` run in a live read-model found all 3
diagnosable real cases were mislabeled error-wedges (a live predicate stuck in
`:error` -- e.g. `missing_url` -- spinning no-op ticks to `max_iterations`),
not genuine budget exhaustion; budgets are hand-authored round numbers with no
data feedback because run-end economics are never persisted; a `max_tokens`
ceiling is silently unbounded when the harness reports no usage. ADR-0058
(extending ADR-0046) decides: persist run-end economics locally, propose
learned budgets from history percentiles, improve dispatch prompting
behavior-first (tools-counter rediscovery signal; opt-in debrief stored as
hypotheses only, never direct prompt mutation; E19/T34.7 benchmark rig as the
only shipping gate), and make budget stops honest (load-time live-config
validation, error permanence taxonomy, live-predicate persistent-error
detection, terminal cause classes, no-usage warning, `max_dispatches`).
Local-first: SQLite read-model only, NO phone-home telemetry (public OSS
repo). Honest-unknown throughout: unreported = NULL/nil, never 0.

Acceptance: wedged runs terminate early as a named `:stuck` (predicate +
reason) instead of `over_budget` at the ceiling; run economics are queryable
from the read-model; `kazi plan`/`adopt` emit provenance-annotated budget
suggestions when history exists; rediscovery candidates and debrief hypotheses
are recorded without ever mutating a dispatch prompt; all shipped, released,
and live-verified on the released binary.
