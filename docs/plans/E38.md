
A twelve-part adoption blog series on the kazi website (`/blog`), working title
**"From Vibe Coding to Reconciliation."** It walks a reader from a vanilla
coding-agent workflow (prompting by feel -- what the industry calls a "vibe coder")
up the same ladder the author actually climbed -- persistent context, knowledge
tiers, real-world verification, structural code understanding, reusable skills, a
written plan, an honest definition of done, safe parallelism -- to a reconciliation
workflow with kazi as the natural conclusion. The goal is ADOPTION through
CREDIBILITY: make the case for kazi's ideas experientially, by having the reader feel
each gap before the product is offered as the close.

Editorial stance is frozen in **ADR-0048** and operationalized in a committed style
sheet (T38.5). The non-negotiables: fellow-traveler tone (never talk down to "vibe
coders" -- it is the honest starting point everyone uses, the author included); every
post independently useful (teach a real, tool-agnostic technique even if the reader
never adopts kazi); credibility over hype (no "10x", no influencer voice, no
unfalsifiable claims); NO vaporware (only shipped behavior; every kazi command
verified against `kazi help --json` / `docs/concept.md` / the ADRs; gated by the
E29/E31 coherence+freshness checks); the product EMERGES (kazi named late). Two rules
added in the second-pass review (ADR-0048 decisions 5, 7, 8): SEPARATE the technique
from the author's PRIVATE stack (lead with the generic, reproducible technique a
reader can do with commodity tools; name a specific personal/internal tool only as
"what I happened to use," with a commodity alternative; never require a private tool
or leak internal hosts/paths/codenames, per ADR-0034); SHOW the loop with diagrams
(T38.20); INSTRUMENT one honest adoption signal (T38.21). It is the LONG-FORM
companion to ADR-0030's short surfaces (README/site), reusing the same hero
transcript, without/with frame, and dogfood "done" gallery -- not a competing
rewrite. Mixed content + engineering: T38.1-T38.4 + T38.20/T38.21 are engineering
(blog infrastructure, visual assets, instrumentation on the Astro site; verify
UC-051), T38.5-T38.19 are content.

**Second pass (skills-coverage review) folded the missing rungs in as SUB-BEATS so
the count stays twelve:** knowledge maintenance (lint/tidy/audit-docs) -> post 3;
safe refactoring + research-as-graph (refactor/ingest/graphify) -> post 5;
adversarial/security review (deep-review/red-team) -> post 8; resilience/recovery
(preflight/resume) + ad-hoc crews vs planned pools -> post 9; reconciliation-applies-
to-work-progress -> post 10; code->prod + deploy triage -> post 4; harden-your-harness
(audit-setup) before unsupervised runs -> post 12. Deliberately EXCLUDED (ADR-0048):
a hierarchical agent-org post, email/lifecycle funnels, a team charter.

The arc (each post ends on the limitation that motivates the next, so the series
reads as one descent into the problem kazi solves):

1. The ceiling of agent-decided "done" -- where prompting-by-feel quietly breaks
   (the agent decides it is finished; there is no objective gate). Sets the thesis
   without naming the product: the bottleneck is not the model, it is the missing gate
   and missing structure. (Builds credibility for concept Gap 1.)
2. Persistent context -- teach the agent to remember your project (a memory file,
   conventions, checkpoints) so it stops re-learning the repo every session.
3. Knowledge tiers (and keeping them honest) -- give decisions and findings a home
   (architecture / decisions / operations / invariants), THEN the hygiene that stops
   them rotting (lint for contradictions, tidy/trim for staleness, audit-docs for
   tier drift). Prefigures kazi's own self-maintaining-docs idea (E31).
4. Real-world verification (and code -> prod) -- give the agent eyes (drive a browser,
   screenshot, exploratory-test) so "green locally" becomes "exercised live," and
   carry it through to production (a release pipeline; triaging a stuck deploy).
   (Builds credibility for kazi's live predicates.)
5. Structural understanding, context economy, and safe refactoring -- stop re-reading
   the whole repo: a code graph for callers/blast-radius, compressing what you feed
   the agent, and reshaping code SAFELY (map dependencies, ordered changes that keep
   the build green). Research-as-graph (ingest/graphify external sources) as a sidebar.
6. From prompts to skills -- codify the good prompt you keep retyping into a reusable
   skill; the compounding move from one-off cleverness to a workflow.
7. Plan the work, then work the plan -- intent as an artifact: a checkable plan,
   tasks linked to outcomes, declared dependencies, instead of an oral to-do list.
8. A definition of "done" that cannot lie (incl. adversarial review) -- the
   credibility centerpiece: tests, coverage, wiring, live probes -- AND an adversary
   trying to break it (security/red-team review) as part of objective done; why "the
   agent said done" is the wrong gate. (Directly sets up kazi's thesis.)
9. One developer, many agents (and recovery) -- parallelism without collisions: pools,
   claims, worktrees; coordinate on resources (blast radius), not identities; plus the
   resilience discipline (preflight before fan-out, resume a halted run) and ad-hoc
   crews vs planned pools. (Builds credibility for concept Gap 2.)
10. The pattern underneath: reconciliation -- the conceptual turn. Every rung above is
    the same loop: declare desired state, drive an agent, check against reality,
    repeat -- whether the "state" is the running system OR your own work-in-progress
    (checkpoint -> drive -> resume). The CI / Kubernetes analogy; truth in the
    controller. Names the CATEGORY before the product.
11. Meet kazi -- the product as the packaging of rungs 7-10 into a controller: declare
    a goal as predicates, the agent drives kazi, it converges or stops honestly
    (`stuck` / `over_budget`). Strictly what it does today; the invocation phrase.
12. Your on-ramp -- recap the ladder, you can start at any rung, how to wire kazi into
    Claude Code, harden your harness first (audit your hooks/permissions before letting
    an agent run unsupervised), an honest "where it is going," a respectful call to try.
