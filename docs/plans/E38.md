
A twelve-part adoption blog series on the kazi website (`/blog`), working title
**"From Vibe Coding to Reconciliation."** It walks a reader from a vanilla
coding-agent workflow (prompting by feel -- what the industry calls a "vibe coder")
up the same ladder the author actually climbed -- persistent context, knowledge
tiers, real-world verification, structural code understanding, reusable skills, a
written plan, an honest definition of done, safe parallelism -- to a reconciliation
workflow with kazi as the natural conclusion. The goal is ADOPTION through
CREDIBILITY: make the case for kazi's ideas experientially, by having the reader feel
each gap before the product is offered as the close.

Editorial stance is frozen in **ADR-0048** and operationalized in a committed style
sheet (T38.5). The non-negotiables: fellow-traveler tone (never talk down to "vibe
coders" -- it is the honest starting point everyone uses, the author included); every
post independently useful (teach a real, tool-agnostic technique even if the reader
never adopts kazi); credibility over hype (no "10x", no influencer voice, no
unfalsifiable claims -- specifics, real transcripts, honest limits); NO vaporware
(only shipped behavior; every kazi command verified against `kazi help --json` /
`docs/concept.md` / the ADRs; gated by the E29/E31 coherence+freshness checks); the
product EMERGES (kazi named late, only once the progression earns it; early posts
name the specific tools the author used as "what I reached for," never "what you must
install"). It is the LONG-FORM companion to ADR-0030's short surfaces (README/site),
reusing the same hero transcript, without/with frame, and dogfood "done" gallery --
not a competing rewrite. Mixed content + engineering: T38.1-T38.4 are engineering
(blog infrastructure on the Astro site, verifies UC-051), T38.5-T38.19 are content
(each `delivers:` a post or asset).

The arc (each post ends on the limitation that motivates the next, so the series
reads as one descent into the problem kazi solves):

1. The ceiling of agent-decided "done" -- where prompting-by-feel quietly breaks
   (the agent decides it is finished; there is no objective gate). Sets the thesis
   without naming the product: the bottleneck is not the model, it is the missing gate
   and missing structure. (Builds credibility for concept Gap 1.)
2. Persistent context -- teach the agent to remember your project (a memory file,
   conventions, checkpoints) so it stops re-learning the repo every session.
3. Knowledge tiers -- give decisions and findings a home (architecture / decisions /
   operations / invariants) so settled choices are not re-litigated and lessons are
   not re-derived.
4. Real-world verification -- give the agent eyes (drive a browser, screenshot,
   exploratory-test) so "green locally" becomes "exercised live." (Builds credibility
   for kazi's live predicates.)
5. Structural understanding + context economy -- stop re-reading the whole repo: a
   code graph for blast radius and callers, plus compressing what you feed the agent,
   makes it cheaper and sharper.
6. From prompts to skills -- codify the good prompt you keep retyping into a reusable
   skill; the compounding move that turns one-off cleverness into a workflow.
7. Plan the work, then work the plan -- intent as an artifact: a checkable plan,
   tasks linked to outcomes, declared dependencies, instead of an oral to-do list.
8. A definition of "done" that cannot lie -- the credibility centerpiece: tests,
   coverage, wiring, live probes; why "the agent said done" is the wrong gate.
   (Directly sets up kazi's thesis -- truth in the controller.)
9. One developer, many agents -- parallelism without collisions: pools, claims,
   worktrees; coordinate on resources (blast radius), not identities. (Builds
   credibility for concept Gap 2.)
10. The pattern underneath: reconciliation -- the conceptual turn. Every rung above
    is the same loop: declare desired state, drive an agent, check against reality,
    repeat. The CI / Kubernetes analogy; truth lives in the controller. Names the
    CATEGORY before the product.
11. Meet kazi -- the product as the packaging of rungs 7-10 into a controller: declare
    a goal as predicates, the agent drives kazi, it converges or stops honestly
    (`stuck` / `over_budget`). Strictly what it does today; the invocation phrase.
12. Your on-ramp -- recap the ladder, you can start at any rung, how to wire kazi into
    Claude Code, an honest "where it is going," a respectful call to try it.
