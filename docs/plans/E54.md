# E54 -- Reliability hardening II: the 2026-07-11/12 execution-sweep bugs (P0/P1)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

A second real-world execution sweep (multi-goal isolated `kazi apply` runs on
1.140.2-1.143.1, 2026-07-11/12) surfaced nine runtime bugs the E53 sweep did
not cover. They cluster into four themes: the **partition worktree+branch
lifecycle** is mismatched with goal-authored `landed` predicates and has a
broken create/reconcile/push/teardown chain that both LEAKS branches (crash)
and DESTROYS uncommitted work (clean-stuck); two **budget-burn safety
regressions** re-open closed guards (#857 child supervision, #769
permission/trust); the `--json` **output contract** silently depends on the
caller's locale; and two **provider/bus DX** gaps swallow diagnostics. Every
future isolated or fleet run hits some of these; three cost real money on a
`stuck`/orphaned run.

Triaged 2026-07-13/14 with three code-grounded root-cause investigations (file
anchors carried per task). Close candidates from the same triage (#1060/#1019/
#1013, fixes already landed) are commented, not closed here.

Acceptance: each fix ExUnit-pinned; a partition run's worktree checks out the
goal's real target branch so an authored branch-name `landed` predicate can
pass; distinct goals never collide on a partition name and a landed/empty
partition branch is cleaned up while unlanded work is preserved; a cleanly-
stuck run's collateral is salvaged before teardown; a launcher killed by
SIGTERM/SIGKILL reaps its whole dispatch tree and `kazi orphans` lists/reaps
detached dispatches; an untrusted-workspace zero-change burn self-diagnoses in
the terminal result (plus a pre-dispatch warning and a fail-fast refusal);
`--json` is valid JSON regardless of locale; a relative-path `custom_script`
`cmd` resolves against the workspace and any exec error surfaces with a reason;
`bus watch` blocks for strictly-new messages instead of firing on backlog.

Kazi clarify gaps (non-blocking): live-target -- these are runtime fixes
verified by ExUnit (+ manual live re-run of the reported scenario where a
fixture cannot reproduce the OS-signal path); no http_probe. scope -- each task
names its seams; per-task goal-files 0066-0074 carry the executable brief.

## Workstream B -- dispatch budget-burn safety regressions (#1073/#1072)

- [ ] T54.5 Parent-liveness monitor reaps a launcher-killed dispatch tree + `kazi orphans` (#1073, regression of #857): the burrito native launcher receives the terminal's SIGTERM; the two #857 mechanisms both watch the BEAM, not the launcher -- the OS watchdog's `parent_pid = System.pid()` is the BEAM (`child_supervisor.ex:116`) which merely re-parents to init and stays alive so the watchdog loop never exits (`child_supervisor.ex:88-95`), and the BEAM-side SIGTERM trap (`finalizer.ex:52-89`) never fires because the launcher does not forward the signal. So `claude -p` runs to natural completion (~$5.13) orphaned. Fix layer A (primary, catches SIGTERM AND SIGKILL of the launcher): a supervised `Kazi.Runtime.ParentMonitor` GenServer started for the CLI that records the launcher pid once (`ps -o ppid= -p <System.pid()>`, reuse the `ChildSupervisor.alive?/1` shell seam at `child_supervisor.ex:143`), polls it ~1s, and on launcher-death runs `Finalizer.record_termination/2` then `System.halt(0)` -- halting the BEAM closes the dispatch port so the existing per-dispatch watchdog reaps `claude` from the correct trigger. Layer B (defense-in-depth, separate fork issue per ADR-0066): burrito launcher forwards SIGTERM/SIGINT to the BEAM. Add a read-only `kazi orphans [--reap] [--json]` verb (the pid registry exists: `run.harness_child_pid` `read_model/run.ex:53`, `RunRegistry.list/0`/`stale?/2` `run_registry.ex:215-274`, the orphan predicate `runtime.ex:1155-1176`); wire the schema entry + `parse_command` clause + handler + docs (T16.4 coherence guard, ADR-0034). ## Workstream C -- output contract + provider/bus DX (#1076/#1096/#1097)

- [ ] T54.7 `--json` is valid JSON regardless of locale (#1076): under a non-UTF-8 locale (`env -i`) `:standard_io` is in `latin1` mode, so `IO.puts(Jason.encode!(payload))` (`cli.ex:1621` `emit/3`, plus direct sites 1636/1724/1731/1750/2834/2900/3013) transcodes each non-ASCII codepoint to the literal 7-char `\x{2014}` -- invalid JSON that fails any strict parser; Jason itself is correct (raw UTF-8). Fix: force ASCII-safe encoding (Jason 1.4.5 supports `escape: :unicode_safe` -> `\uXXXX`, verified locale-independent) via one private `encode_json!/1` helper routed through `emit/3` and every direct `Jason.encode!` write site; do NOT mutate global `:io.setopts` (process-global, fights `env -i`). Human output (`emit(false, ...)`) untouched. - [ ] T54.9 `bus watch` blocks for strictly-NEW messages, not backlog (#1097): `watch/1` (`bus.ex:167`) does an initial `drain(conn, opts, true)` that pulls all pending durable messages -- including backlog a prior `peek` left un-acked -- so it fires immediately (~3s) on already-seen messages instead of blocking, degenerating into `peek` and making every wake a false positive (defeats the issue #1091 no-poll purpose).
