### E40 -- Behavior specs: wire the dormant Gherkin importer into a first-class `docs/specs/` tier (P2, ADR-0050)

kazi has no doc tier that says "here is the behavior being built" -- ADRs are WHY,
`docs/plan.md` tasks are one-line labels, `goal.toml` predicates are the machine
check. ADR-0050 closes this by adding `docs/specs/` (Gherkin `.feature` behavior
specs) as a sixth tier and wiring the ALREADY-SHIPPED, ALREADY-TESTED
`Kazi.Reconcile.GherkinImporter` (ADR-0021/T13.2) into the CLI so predicates are
DERIVED from a reviewed spec instead of hand-typed. `GherkinImporter` is currently
referenced nowhere outside its own module + test file -- this epic is what makes it
load-bearing.

Acceptance: a kazi contributor can write a `docs/specs/<slug>.feature` behavior
spec, run it through a CLI verb that upserts its Scenarios as `[[predicate]]`
entries in a goal, reference the spec from its WBS task via an optional `spec:`
field, and have the spec archive alongside its epic when the epic archives
(ADR-0036 L1). No new grammar, no new predicate provider -- only CLI wiring, a
doc-tier definition, and archive-path glue over existing, tested code.
