# E65 -- Deterministic session/team identity: derived teams, daemon-assigned names, UUID-keyed presence (#1430, P1)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

Operator report (David, 2026-07-18; issue #1430 carries the full design and
the four observed failure modes): session/team identity is three loosely
coupled free-form strings (session UUID, friendly name, team string) chosen
at different times by different actors. Observed live in one day: THREE
variants of one team string splitting @team delivery; every friendly name
dropped on each daemon restart; renames minting duplicate presence entries;
and no harness-provided human-readable name in the environment (only
`CLAUDE_CODE_SESSION_ID`).

Design principle (operator-selected via four decisions, recorded on #1430):
**identity is the immutable session UUID; teams are DERIVED from the repo's
git origin; names are DAEMON-ASSIGNED labels bound to the UUID, durable in
JetStream KV; rename is an alias update on the one UUID-keyed row with a
tombstone-alias grace window.** Scope: kazi core only -- no harness wrapper.

Exit criteria: two sessions running `kazi bus join` argless from different
checkouts (or machines) of the same repo land in the SAME team with zero
typed team strings; a daemon restart preserves every name binding; a rename
never creates a second presence row and in-flight `tell <old-name>` still
lands during the grace window; the leading-dash team-string class is
structurally impossible (slugs carry a fixed prefix).
