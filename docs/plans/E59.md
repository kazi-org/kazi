# E59 -- Concurrency, multi-process safety, and CI flakiness (P1)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

Five issues found via a full open-issue triage pass (2026-07-17), all in the
"more than one thing running at once" family -- the standing assumption that
multiple kazi processes (or multiple test runs, or multiple CI jobs) share
state and must not corrupt each other. Two (#1019, #1255) are ALREADY
PARTIALLY FIXED per their own triage comments, left open pending live
verification -- this epic closes that verification gap rather than
re-investigating from scratch. Two (#1025, #1186) are CI test-flake clusters
that may share a root cause -- scoped as one investigation feeding two
possible fix tracks rather than assumed-separate from the start. One (#937)
is a large, partially-addressed proposal that needs its full gap inventory
read before scoping -- deliberately NOT pre-decomposed here beyond the first
task, per the rolling-wave depth policy.

Acceptance: the serial `kazi apply` path either has documented isolation
parity with `--parallel`, or the remaining gap is explicitly scoped as
follow-up tasks with named owners; #1019's migration-lock fix holds under a
REAL mixed-version release-window reproduction, not just code review;
#1025/#1186's flaky tests are root-caused (shared cause or not) and fixed
without weakening any real assertion; #1255's original `kazi version` hang
trigger is either reproduced and fixed, or downgraded to an explicit
monitoring/detection task with a stated reason it could not be reproduced.
