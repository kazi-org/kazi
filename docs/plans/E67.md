# E67 -- Dashboard velocity surface: per-agent delivery, tokens, and stuck stats, full-fusion (P2)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

Operator request (2026-07-18): "see the velocity of my Claude agents --
delivered tasks, tokens consumed, stuck stats" -- as part of the DASHBOARD,
built right once, not as scripts. Scope decision recorded (operator,
2026-07-18): FULL FUSION -- the surface covers BOTH agent populations:
(a) kazi-driven runs, already instrumented in the read-model (iterations,
verdicts, per-run token KPIs, `kazi economy` percentiles), and (b) the
operator's interactive harness sessions, whose activity today lives only in
local session transcripts outside kazi. That requires new plumbing: a
session->run->commit linkage (the E65 durable session identity joined to the
session-trailer already present on every commit) and a session-stats
collector that ships AGGREGATE counters (never transcript content) into the
read-model.

Design constraints, non-negotiable: every displayed element maps to an
objective source already in the read-model (ADR-0011 read-only projection);
all velocity displays are RATES and RATIOS, never dates or completion
estimates (ADR-0046 honest-unknown); all new writes route through the daemon
single-writer path (ADR-0068); transcript CONTENT never leaves the machine --
only aggregate counters (token counts, event counts, timestamps) are
ingested, and the collector is opt-in per machine.

Exit criteria: the dashboard answers "how fast are my agents moving" with
per-agent and fleet-level delivered-task rates, tokens consumed per delivered
unit, and stuck/rescue ratios, across both populations, live on a released
binary with real fleet data; every number traceable to its source; zero
fabricated time estimates anywhere on the surface.
