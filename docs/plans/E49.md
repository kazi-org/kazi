# E49 -- Scenario predicates: capability-level verification by demonstrate-then-pin (P1, ADR-0064)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

The catalog verifies code (`:tests`/`:static`/...), artifacts (`:cli`, planned
E43), and surfaces (the E43 `:browser` assertion pack), but not CAPABILITIES:
"a user can create and download a PAT" has no machine-checkable form. ADR-0054
d3 lowers `@interface:web` Scenarios to `:browser` predicates, but nothing owns
turning prose steps into executable `steps`/`assertions`. ADR-0064 (Accepted
2026-07-08) decides: a first-class `scenario` provider binds one tagged Gherkin
Scenario to a committed PIN -- a replayable trace expressed in the EXISTING
`:browser`/`:cli` config vocabulary -- and the predicate passes ONLY when the
pin validates (every When -> >=1 step, every Then -> >=1 assertion,
Scenario-hash current) and replays green through the delegated surface
provider. Pins are authored/repaired by a DEMONSTRATOR dispatch role
write-disjoint from the fixer (role-scoped ADR-0042 `read_only_paths`), so
agentic nondeterminism is quarantined at authoring time and evaluation stays
deterministic. Repin-with-diff distinguishes selector rot from capability
regression; repeated demonstration failure is a named stuck cause, never a
loop. Hand-authored pins work day one (Wave A ships value with zero new
dispatch machinery).

Pin artifact (kazi-owned, versioned; full reference authored in T49.1 and
documented in T49.5): one JSON file per Scenario at
`docs/specs/pins/<derived-id>.pin.json` (the importer's derived id), fields
`pin_version` (=1), `spec`, `scenario`, `scenario_sha` (SHA-256 of the
normalized Scenario text), `surface` ("browser"|"cli"), `minted` (`%{commit}`),
`inputs` (`%{name => generator}`), `trace` (VERBATIM surface-provider config:
browser `steps`+`assertions`, or the T43.8 cli script matrix), `map` (per
Gherkin step: `%{step, steps: [trace indices], assertions: [indices]}`).
Pin-state classification enum used across the epic: `:pinned` (valid + hash
current) | `:unpinned` (file absent) | `{:stale, :spec_changed | :code_drift}`
| `{:invalid, [reasons]}`. Truth invariant (ADR-0064 d1): `:pass` comes ONLY
from a pinned replay; unpinned/stale/invalid are `:fail` (routed work), a
missing SPEC file or unreadable workspace is `:error`.

Acceptance: a goal can declare `provider = "scenario"` against a
`docs/specs/*.feature` Scenario and converge only when its committed pin
validates and replays green through the browser (or, once E43 lands, cli)
provider; a hand-authored pin satisfies it with no demonstrator involved; with
the demonstrator enabled, an unpinned scenario gets its pin authored by a
write-disjoint role whose output is accepted only on validate-plus-green-replay;
a red pin after code drift re-demonstrates (auto) or parks for the operator
(manual) with the repin diff as evidence; repeated demonstration failure on an
unchanged workspace terminates as `:stuck` cause `capability_unreachable`; all
shipped with docs, schema entries, and a dogfood pin over kazi's own surfaces.
