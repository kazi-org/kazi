Operator directive (David, 2026-07-18): a kazi subcommand that answers "where

proposal rows with no headline, no percentages, no blocked bucket, no
summaries. So this epic is the DELTA that makes the existing verb answer the
question rather than dump the ledger. Related issues found: #1160 (v1, both
surfaces shipped -- close on evidence when this lands), #1070 (cost line,
T60.5 in flight -- joins the sitrep once landed), #1156/#1154 (adjacent,
shipped). No new verb: v2 upgrades `kazi portfolio` in place, `--json`
additive, schema_version stays 2.

Bucket semantics (the five David named, mapped to objective sources per
ADR-0011 -- every bucket a read-model/DAG projection, nothing hand-set):

- **planned** = proposals status=proposed (awaiting approval).
- **todo** = proposals status=approved with NO registered run (ready to
  dispatch) -- v1 lumps these into planned; the split is the actionable line.
- **in progress** = registry runs, fresh heartbeat, status running.
- **blocked** = three causes, each entry NAMING its blocker: (a) roadmap-DAG
  blocked -- a `needs` dep unconverged or poisoned (DagSnapshot `:blocked`
  already computes this); (b) stuck -- naming the persistently-failing
  predicate slice (the run's last vector); (c) over_budget -- naming the cap.
- **done** = terminal converged (landed when integration data exists).

Exit criteria: `kazi portfolio` opens with ONE headline line of counts +
percentages across the five buckets; each bucket renders a bounded summary
(top-3 + "+N more", `--full` restores the ledger); every blocked entry names
its blocker; "how is it going" is honest rate data (predicates green/total,
red->green movement over recent iterations) and NEVER a projected date
(ADR-0046); v1 `--json` consumers are byte-unaffected.
