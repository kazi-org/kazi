### E42 -- Fix kazi's self-teaching artifacts: no personal-skill assumptions, retire dead Graphify retrieval (P1, ADR-0052)

An audit found `AGENTS.md` and the `kazi install-skill`-generated `SKILL.md`
(both shipped to every kazi user) assume the operator's personal Claude Code
skills (`/plan`, `/tidy`, `/loop`, `/qualify`) are a universal baseline -- they
are not. The same audit found `Kazi.Retrieval.Graphify` shells out to a
`graphify` CLI that does not exist (the real `/graphify` is an agent-executed
skill, not an installable binary), so that "real" backend has never worked for
anyone. This epic fixes both: rewrites the teaching artifacts to describe the
upstream/downstream CONTRACT generically instead of naming external skills,
adds a coherence guard so this class of mistake cannot silently recur, and
retires the dead retrieval adapter while keeping the pluggable seam intact.

P1: this is a correctness bug in already-shipped, released guidance, not a new
feature -- every kazi adopter following `AGENTS.md` or the installed SKILL.md
today hits a dangling reference.

Acceptance: neither `AGENTS.md` nor the generated `SKILL.md` contains a bare
`/<word>` token that isn't a real `kazi` CLI verb; the coherence test enforces
this going forward; `Kazi.Retrieval.Graphify` is gone; `Kazi.Retrieval`'s
pluggable seam + `NoOp` default still work exactly as before.
