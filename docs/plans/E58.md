# E58 -- Bus reliability: version-skew dead-letters + CLI discoverability (P1)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

Two real bus bugs found via a full open-issue triage pass (2026-07-17). Both
are trust bugs in the E51/E55 session-bus surface (already in daily
production use across two machines): one is a silent, ASYMMETRIC failure
(writes succeed, reads fail) that makes a directed inbox message
disappear without any error surfacing to the sender -- the worst possible
failure shape for a coordination primitive, because it trains operators to
trust an ack that lied. The other is a CLI usability gap (no in-CLI way to
discover `bus post`'s required `<kind>` argument or valid kind values,
plus housekeeping log lines polluting stdout) that has already cost a
reporter real trial-and-error time.

Acceptance: a CLI/daemon version-skew condition either fails loud on write
too (not just read) or is detected and reported before it can produce a
dead-lettered message; `kazi bus post` without `<kind>` either defaults
sensibly or fails with a usage line enumerating valid kinds; every `bus`
subcommand has real per-verb `--help` text; `kazi bus read` stdout carries
only message payloads, never housekeeping log lines.

## Workstream A -- version-skew dead-letter (#1227)

- [ ] T58.1 Investigate the bus RPC/version-negotiation path for the read/write asymmetry (#1227): a long-running bus daemon (started under an older kazi) kept accepting `bus join`/`who`/`tell`/inbox-accumulation from a newer CLI (1.172.0), but `bus read`/`bus read --peek` from that same newer CLI failed with `error: daemon could not read the bus: unknown_op`. Net effect: `who` showed the session with a nonzero inbox count, senders got a successful `told <name>` ack, but the recipient could never read any of it -- a silent dead-letter queue where senders believe delivery succeeded. Read the daemon<->CLI RPC dispatch code (the `unknown_op` error implies an operation-code mismatch between CLI and daemon protocol versions) and determine WHY writes succeed but reads fail under this skew -- is it that the read path added a new op code after the write paths were already stable, or something else. Write findings to docs/devlog.md before scoping the fix in T58.2 (this task is investigation-only; do not fix yet, the fix approach depends on the root cause). - [ ] T58.2 Fix the version-skew asymmetry per T58.1's finding (#1227): implement whichever fix direction T58.1 recommends. ## Workstream B -- bus CLI discoverability (#1060)

- [x] T58.3 `kazi bus post` kind UX + per-verb help + clean stdout (#1060): three related gaps in the same issue thread. (1) `kazi bus post "some text"` (without the required `<kind>` positional arg) fails with an unhelpful error that does not list valid kinds -- either default `kind` to `fact` when omitted, or fail with a usage line enumerating every valid kind (pick whichever preserves the existing explicit-kind behavior for callers who already pass one; do not silently change behavior for existing correct callers). (2) `kazi bus post/read/tell/who --help` (and other `bus` subcommands) currently print the generic top-level `kazi` usage block instead of subcommand-specific usage -- add real per-verb `--help` text for every `bus` subcommand, following whatever per-command help-text convention the rest of the CLI already uses (check `lib/kazi/cli.ex`'s `@flag_docs`/command help pattern -- this repo has a known landmine where a new flag/command needs registering in FOUR places: `@switches`, the command's flag list, the help TEXT, and `@flag_docs`, or `help --json` crashes; run the help-schema/coherence test suites before pushing regardless of whether this task touches flags). (3) `kazi bus read` currently intermixes housekeeping log lines (e.g. "Skipped cleanup of older version (vX.Y.Z): still in use by a running process") with message payloads on stdout, which breaks machine parsing of `bus read`'s output -- route housekeeping/log lines to stderr so stdout stays clean JSON/text for scripting.
