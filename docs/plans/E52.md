# E52 -- The daemon becomes the single writer for the read-model (P2, ADR-0068)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

ADR-0068 (Accepted 2026-07-10): with the E51 daemon running, it is the only
process that opens `~/.kazi/kazi.db` read-write -- clients write via the
daemon's versioned Unix-socket API, only the daemon migrates, and version skew
becomes an explicit handshake ("restart daemon or continue without
persistence") instead of a migration-lock fight. Eliminates the #1019 class
(mixed-version writers, deadlocks, Guard-blind runs) structurally. Reads may
stay direct (WAL); no daemon = today's behavior unchanged. The daemon also
hosts ADR-0005's ETS live layer for the dashboard/starmap.

Exit criteria: two kazi versions running concurrently through one daemon
never deadlock and never run persistence-blind without an explicit operator
choice; the duplicate-run guard and economics stay reliable during a release
window; rollback (`no daemon`) preserves today's semantics byte-for-byte.

## The write surface (T52.0 inventory)

Every read-model write goes through `Kazi.Repo` (the single `Ecto.Repo`,
`Ecto.Adapters.SQLite3`, `lib/kazi/repo.ex`). The exhaustive grep for
`Repo.insert|update|delete|insert_all|update_all|delete_all|transaction|
insert_or_update|insert!` plus the raw `Repo.query!` FTS writes finds **22 write
call sites across 6 modules**, which group into **~20 logical write entry
points** (the public functions a client actually calls -- these are the surface
the daemon's `write` op must cover):

- `lib/kazi/read_model.ex` (9): `record_iteration/1` (insert), `record_debrief_hypotheses/1` (insert), `transition_proposed_goal/3` (update), `propose_memory/1` (insert), `transition_proposed_memory/2` (update), `put_cached_pack/4` (insert), `invalidate_cached_pack/1` (delete_all), `put_cached_snippets/5` (insert), `invalidate_cached_snippets/1` (delete_all).
- `lib/kazi/read_model/run_registry.ex` (6): `start/1` (insert), `heartbeat/1` (update), `record_harness_session/2` (update), `record_harness_pid/2` (update), `finish/3` (update), `record_termination/2` (update) -- the duplicate-run guard + economics rows, the #1019 blind-run class ADR-0068 targets.
- `lib/kazi/read_model/run_reaper.ex` (1): `reap/0` (update).
- `lib/kazi/read_model/pause_checkpoint_store.ex` (2): `put/1` (insert_or_update), `delete/1` (delete_all).
- `lib/kazi/authoring.ex` (1): the `ProposedGoal` insert with `on_conflict:` (an UPSERT, not a plain insert -- the write op must carry conflict options).
- `lib/kazi/memory/semantic_index.ex` (3): `upsert_hash/3` (`Repo.insert!`) plus two raw `Repo.query!` FTS statements (a `DELETE FROM memory_chunks_fts` + an `INSERT`) that together are ONE logical upsert. LANDMINE for the write op: this surface is NOT all Ecto changesets -- it includes multi-statement raw SQL, so the daemon's write handler applies an opaque, pre-serialized write plan (changeset + options, or a raw statement list) atomically, never a "changeset only" shortcut.

Reads are out of scope (ADR-0068 point 4 -- WAL supports concurrent readers; `kazi status`/`economy` keep opening read-only). Only the 20 write entry points centralize.

## The write API over the SHIPPED T51.1 control socket

The design rides the real, shipped T51.1 seams (verified on main), not a
hypothetical protocol:

- **Transport.** The daemon's control plane is `Kazi.Daemon.Listener` binding a Unix socket in `packet: :line, active: false` with `buffer: Kazi.Daemon.Probe.socket_buffer()` (= **1_048_576 bytes**, `lib/kazi/daemon/probe.ex`), and a 5000 ms `:gen_tcp.recv` bound in `handle_conn/4`. The client speaks it through `Kazi.Daemon.Probe.request/2,3` (`connect` -> `send(Jason.encode!(payload) <> "\n")` -> `recv` with a caller-supplied `recv_timeout`). Both ends bind the SAME buffer; L-0052 (docs/lore.md): `packet: :line` truncates an over-long line SILENTLY on the receiving end, so every write request AND reply MUST stay under `socket_buffer/0` -- the write op enforces that bound explicitly (see batching) rather than inheriting a silent-corruption bug.
- **The op.** A new `write` op joins `ping`/`read`/`shutdown` in `Kazi.Daemon.Control.handle/2` (its moduledoc already invites this: "a later task can add ops here without touching the accept loop"), delegating to a new server-side `Kazi.Daemon.Write` -- the exact shape of `read` delegating to `Kazi.Daemon.BusRead`. Request: `{"op":"write","schema_vsn":<int>,"batch":[{"kind":"record_iteration","plan":<opaque>}, ...]}`; reply `{"ok":true,"applied":N}` or `{"ok":false,"error":"<reason>"}` -- the same `{ok, ...}` / `{ok:false, error}` envelope `BusRead` already returns.
- **Batching.** The socket is one-line-request / one-line-reply, so batching coalesces several logical writes into one `batch` array to amortize connect + round-trip. `Kazi.Daemon.Write` applies the whole batch inside ONE `Repo.transaction` (transaction is already in the write inventory) so a batch is atomic. The client caps a batch so the encoded line stays under `socket_buffer/0` with headroom (the digest already runs to tens of KB against a 1 MiB buffer, so a bounded write batch fits comfortably); a batch that would exceed the buffer is SPLIT client-side, never sent -- mirroring `BusRead.refuse_full/1`, which refuses the one unbounded payload rather than let `packet: :line` truncate it.
- **Backpressure.** The daemon is the single writer with one `Repo` pool; writes serialize through it. `Kazi.Daemon.Write` applies a request synchronously (the `Repo` pool does its own queueing) within the client's `recv_timeout`; a bounded in-daemon inflight cap returns `{"ok":false,"error":"busy"}` rather than growing an unbounded queue, and a `busy`/timeout is a clean client-side degrade (fall back to the version-stamp-and-refuse path, never a blind write). This reuses `Kazi.ReadModel.Guard`'s never-hang discipline: a write that cannot be confirmed within budget degrades explicitly, it never hangs the caller.
- **The version-skew handshake.** The load-bearing version is the SCHEMA/migration version, not the app `vsn`. `Kazi.ReadModel.Migrate` already computes `binary_version/1` (the highest `priv/repo/migrations/<ts>` timestamp this binary ships) and stamps it into the `kazi_schema_meta` single-row table (`db_stamped_version/1`). The `ping` reply (which already carries `vsn`) gains an additive `schema_vsn` field (the daemon's stamped version); an old client ignores it. On connect the client classifies skew exactly as `Migrate.migrate_or_refuse/2` already does: client schema == daemon -> write through; client schema < daemon -> the API is additive within a major, so the daemon accepts the old client's write (ADR-0068 point 3); client schema > daemon (the daemon is OLDER -- the common mid-release-window case) -> the daemon refuses and the client surfaces the single-line operator choice "daemon is older than this client; restart it (`kazi daemon restart`) or continue without persistence" (ADR-0068 point 3, the existing `{:error, {:newer_schema, ...}}` verdict moved to the handshake).

## Migration ownership + no-daemon fallback (decisions)

- **Only the daemon migrates (cutover).** Today `Kazi.CLI.migrate_read_model/0` (`cli.ex:7494`, via `Kazi.ReadModel.Migrate.run/2`) migrates at EVERY CLI boot -- N binaries x M schema versions on one file, the #1019 cause. Cutover: when a daemon is up, the CLI/run process never opens the file read-write and SKIPS migration entirely; the daemon migrates once, at its own startup, BEFORE its write server accepts writes (a new supervisor ordering: `Migrate.run/2` -> then start `Kazi.Daemon.Write`). Absent a daemon, `migrate_read_model/0` runs exactly as today (ADR-0068 point 5, byte-for-byte rollback).
- **No-daemon fallback hardening (version-stamp-and-refuse).** ADR-0068 keeps stamp-and-refuse as the no-daemon hardening (rejected as the PRIMARY design because it maximizes blind runs, kept as the fallback). Today the refusal fires only at migrate time (`migrate_or_refuse/2`'s `{:error, {:newer_schema}}`). E52 extends it to WRITE time: a direct-writing client whose `binary_version` < the file's stamped `schema_vsn` refuses each write with the one-line operator degrade and runs persistence-blind VISIBLY (a Guard-style explicit degrade), so an older binary never writes blind against a newer schema even with no daemon present. This is the structural improvement over today, where an older binary that predates a Guard path could hang or write against a schema it does not understand.

Deliberately kept OUT of E52 (ADR-0068 point 6, a separate concern): hosting
ADR-0005's ETS live layer in the daemon for the dashboard/starmap. That is a
read/serve path; E52 centralizes WRITES. It gets its own task when the
dashboard's live-source work (E55's T55.3 transport source) needs it.

## Waves

- **Wave E52-A (2 parallel):** T52.1 (the client `Writer` seam), T52.2 (the `schema_vsn` handshake). Both are additive and route no writer yet -- the two independent foundations, different files entirely.
- **Wave E52-B (2 parallel):** T52.3 (the `write` op + `Kazi.Daemon.Write` server), T52.4 (migrate-before-serve + `daemon restart`). T52.4 depends on T52.3 (the write server it orders); run T52.3 first, then T52.4, or overlap once T52.3's server module exists.
- **Wave E52-C (4 parallel):** T52.5 (route the 20 writers), T52.6 (CLI migration cutover), T52.7 (no-daemon stamp-and-refuse hardening), T52.8 (client-newer skew degrade). T52.5 and T52.6 share R-cli-ex/R-read-model discipline; T52.7 depends only on T52.2 and may start earlier if a session is free.
- **Wave E52-D:** T52.9 closes the epic post-release, on two live versions through one daemon.

**Ordering constraint that is not a dep edge:** do NOT route any writer (T52.5)
before the write server (T52.3) AND its migrate-before-serve wiring (T52.4) are
both green -- routing a write to a daemon that has not migrated would hand a
client a "no such table" where today it gets a clean direct write. T52.5 carries
the dep edge; this note carries the discipline.

**Shared-file locks.** `lib/kazi/cli.ex` (R-cli-ex, touched by T52.4/T52.6) and
the read-model writer modules (R-read-model, touched by T52.1/T52.5/T52.7/T52.8)
are the contended files; the two sub-waves that touch them serialize on the
lock, everything else runs free.
