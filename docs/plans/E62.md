# E62 -- Gherkin runtime predicate provider + parallel-landing follow-ups (P1)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

Two issues found via a full open-issue triage pass (2026-07-17).

Issue #1107 wants a `provider = "gherkin"` RUNTIME predicate provider --
different from the already-shipped E40 (`kazi spec import`, an author-time
CLI verb that scaffolds `custom_script` predicates). The design was fully
hashed out LIVE with an external stakeholder (the Sire team, a real
production consumer with 42 `.feature` files + 25 goal-files ready as
acceptance fixtures) across the issue's comment thread, and is ALREADY
LOCKED in `docs/adr/0071-gherkin-runtime-provider.md` (Accepted) -- this
epic implements that already-decided design, it does not redesign it. Read
the ADR in full before starting; every edge case (Scenario Outline handling,
missing-scenario semantics, identity matching, verdict source) is already
decided there.

Issue #1241 is an explicitly deferred follow-up to T44.10 (PR #1240, the
per-group branch/commit/push contract for `--parallel` + `[integration]`
runs) -- two pieces T44.10 intentionally left out because they carry
independent scheduler regression risk that deserved dedicated review rather
than riding along in T44.10's PR.

Acceptance: a goal-file predicate with `provider = "gherkin"` expands at
load time into one sub-predicate per Scenario (grouped under the Feature),
ingests real per-scenario verdicts from ANY runner via cucumber-json (stdout
or file), and Sire's `storage-store.feature` reconciles natively as the
first proof; `--parallel` per-group landed results get live origin-branch
verification and persist into `kazi status` output.
