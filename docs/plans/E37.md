
Acceptance: `kazi apply <goal> --harness gemini_cli [--model <m>]` drives Google's
Gemini CLI (`gemini`) the same way it drives `claude`/`opencode`/`codex` -- added
as PROFILE DATA, not a new module (ADR-0016), meeting the conformance contract
(ADR-0022): non-interactive single-prompt, machine-parseable stdout, correct under
a non-TTY subprocess. This closes the one named coverage gap (the registry already
forward-references `:gemini_cli`); it does NOT pursue cross-harness feature parity
(non-Claude profiles still forward only `:model`, by ADR-0016 design). Researched
contract (`gemini` v0.38.2, `gemini --help`): **non-interactive** `gemini -p
"<prompt>"`; **native JSON** `-o/--output-format json` (also `stream-json`); **model**
`-m/--model <m>`; **auto-approval** for headless action-taking via `--approval-mode
yolo` (or `-y/--yolo`) -- the analogue of Antigravity's `--yes`, required because a
non-interactive harness must take tool actions without a prompt; **auth**
`GEMINI_API_KEY` (or Google OAuth / Vertex `GOOGLE_API_KEY`), supplied by the
operator's environment. Because `gemini` has first-class `-o json`, it is a
FULLY-CONFORMANT addition (the Codex template `profiles/codex.ex`), not a workaround
case like Antigravity -- UNLESS the live smoke (T37.2) finds a non-TTY stdout defect,
in which case it drops to `prompt_via: :file` like Antigravity. Each profile is
unit-tested (`build_args`), golden-transcript-tested (`parse` against recorded real
output), and live-smoked behind an excluded `:gemini_cli_live` tag -- the same recipe
(`docs/add-a-harness.md`, T14.1 helper) that added codex/antigravity/claw.
