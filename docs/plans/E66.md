SUPERSEDED DIAGNOSIS (2026-07-20): the original three hypotheses in this task -- "the isolated pass is not killed", "the tick never fires under burrito", "the :error log never reaches the launchd log file" -- are ALL DISPROVEN; do not re-run that elimination. The live run on v1.273.0 (ancestry-verified) with the tick-lifecycle counters gave `ticks_fired: 5, passes_completed: 1, passes_crashed: 3, passes_killed: 0` and `velocity pass went DOWN without completing (:ehostunreach) -- not a deadline kill`: the tick DOES fire, the #1603 kill path is NOT what ends these passes (passes_killed stayed 0), and the log DOES reach the file. See issue comments 5025868787 and the correction 5025957582. ACTUAL ROOT CAUSE: the pass's only network call -- the collector's best-effort bus `fact` post -- routed through `Kazi.Bus.run/3`, which used `Task.async` and therefore LINKED; against an unreachable NATS host the linked worker's exit killed the pass (a bare `spawn_monitor`'d process that does not trap exits), uncatchable by the collector's own try/rescue. The separate hang ("velocity pass still running after 10000ms") is the SAME root cause with the network blackholing instead of failing fast: `run/3`'s 15s bound was paid once PER SESSION, overrunning the tick while staying under the 120s kill deadline. Fixed by PR #1637 (unlink `Bus.run/3`; circuit-break the fact post after one bus failure per pass), on top of PR #1612 (bounded transcript scan), #1619 (tick-lifecycle counters that made this diagnosable) and #1629 (a caught-up pass no longer resets `last_session_count` to 0 -- which had made a completed pass report a false "0 session(s)"). REMAINING: the live confirmation only.
