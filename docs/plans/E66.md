have no plan home. Two are a release-binary boot family: #1483 is a LIVE P0 on
the current release (main == 1.243.0) -- the burrito standalone entry hands off

that owns `Kazi.Repo` is stood up, and while CLI verbs recover via
`ensure_read_model`, the long-lived daemon/dashboard path has no such call, so
`MissionControlLive` raises "could not lookup Ecto repo Kazi.Repo" (dashboard
500s, no read-model persistence). #1484 (launchd-managed `bushost` exits 78
silently) is the same family: the release binary booting as a long-lived
process under a minimal environment. #1406 is the approved `swift_test`
provider proposal, which already has real implementation commits in a stalled
`kazi apply` internal worktree but no plan task, so pool sessions cannot pick
it up.

Acceptance: the released binary's daemon/dashboard path serves with a started,
migrated read-model (no Repo-lookup raises); a launchd-managed bushost either
runs or fails LOUD with output; `swift_test` is a first-class predicate
provider where 0-tests-matched is red; each fix carries its regression test.
