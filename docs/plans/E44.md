### E44 -- Landing is part of convergence: `[integration]` + implicit `landed` predicate + controller-owned process contract (P1, ADR-0055)

The sharpest regression operators hit moving from a prescriptive orchestrator to
`kazi apply`: **converged goals leave dirty worktrees -- nothing commits**.
Root cause is structural: `Kazi.Loop.decide/2` reaches `:converged` (clause 1)
before `:integrate` (clause 3) whenever a goal's predicates are all code-kind, so
landing only happens when a live predicate keeps the vector unsatisfied. Two
compounding gaps: `Kazi.Actions.Integrate` bulk-commits (`git add -A` + one
monolithic commit), and the dispatch prompt carries zero process discipline
(no commit-as-you-go, zero-stub, lore-grep, migration-number safety). ADR-0055
fixes all three the kazi-native way: landing becomes part of the OBJECTIVE bar
(an implicit `landed` predicate synthesized from a new `[integration]` goal-file
block), Integrate verifies-then-ships instead of committing for the agent, the
working discipline moves into a versioned controller-side process-contract prompt
section (goal-files stay declarative -- the ADR's explicit decision on the
"embed the ~15 discipline blocks into kazi plan?" question is NO), `--parallel`
groups land as per-group branches/PRs merged in `needs` order with `git cherry`
silent-revert verification, three deterministic gate providers
(`no_stubs`/`oss_hygiene`/`docs_updated`) make the external wave gates
kazi-native, and `kazi apply` grows a preflight. This is also the missing half of
ADR-0031's subsumption claim: `kazi apply --parallel` cannot replace an external
pooled orchestrator while the pool lands PRs and kazi does not.

Existing seams reused: the goal loader section builders (`build_scope`/
`build_enforcement` pattern in `lib/kazi/goal/loader.ex`), the predicate vector +
T0.8 termination guard (UNCHANGED -- `landed` is just another predicate),
`Kazi.Actions.Integrate` + its injectable `:integrator` seam (tests never touch
GitHub), the T19.1 orientation-prefix prompt section (the process contract is one
more stable cacheable section), the `needs`-DAG scheduler (ADR-0027/0028) whose
topological order IS the merge order, the `[harness]` permission wiring (#776) and
`permission_denials` surfacing (#769), and the E29 leak-guard scan logic for
`oss_hygiene`.

**Waves.** Wave A is the core chain: T44.1 (schema) -> T44.2 (implicit predicate)
-> T44.3 (Integrate rewrite); T44.4 (process contract) and T44.5 (permission
alignment) join after T44.1. Wave B (gate providers T44.6-T44.8, preflight T44.9)
is independent of Wave A and runs in parallel with it. Wave C (parallel landing,
T44.10-T44.11) needs Wave A's core. Wave D (authoring floor T44.12, self-teaching
docs T44.13, dogfood T44.14) closes the epic.
