### E41 -- Crystallize discovered truth: Gherkin + tags for product-level use cases, iterative discovery via `kazi init --discover`, and prod-log predicate correlation (P2, ADR-0054/ADR-0051)

`I \ A` / `A \ I` (ADR-0021) has three declared `I`-sources (OpenAPI/gherkin,
prose, behavior specs) but no empirical one. An earlier draft of this epic
invented a bespoke `usecase-manifest.json` schema and a one-shot `kazi spec
discover` command -- corrected by ADR-0054: that repeated the exact mistake
ADR-0050 (one epic earlier) had just avoided by choosing Gherkin over a new
grammar, and it had zero producers anywhere. This epic instead (a) extends the
ALREADY-SHIPPED `Kazi.Reconcile.GherkinImporter` to read Cucumber tags
(`@role:`/`@priority:`/`@interface:`) so a directory of tagged `.feature` files
at the product/capability scope IS the use-case catalog, (b) adds an iterative
`kazi init --discover` convergence (reusing the shipped surface-scanner, T13.4)
instead of a one-shot prompt, and (c) an unrelated, independent prod-log
correlation provider extension (ADR-0051 decision 4, retained as-is).

Acceptance: a repo that already has Cucumber `.feature` files imports them via
the EXISTING `kazi spec import` verb with no new flag; `kazi init --discover`
writes a starter goal that converges toward "every surface-scanned element has
a Scenario" via ordinary `kazi apply`, no external skill and no one-shot
exhaustiveness assumption required; a predicate can opt into prod-log
correlation so a false-green result gets flagged.
