### E43 -- Higher-level interactive-surface predicates: a `:browser` assertion pack + a first-class `:cli` provider (P1, ADR-0053)

ADR-0043 grew the catalog along the code-side and live-telemetry axes but left the
**surfaces a user actually operates** thin: `:browser` knows only `visible`/`text`,
and NO predicate proves the shipped BINARY runs (only `:tests`, which proves the code
compiles). This epic closes both gaps per ADR-0053: (A) extend the Playwright runner's
assertion vocabulary with a UI pack -- console-clean, a11y, visual, form-validation,
richer DOM assertions, a viewport matrix -- which is runner + schema only, zero
kazi-core change (the ADR-0040 dividend); and (B) promote `:cli` to a first-class
provider that drives the built binary as a user invokes it (exit-code + stdout/stderr
assertion matrix + a `--help` golden-snapshot mode), dogfooded against kazi's OWN
release binary to catch the recurring release-boundary regressions (`:noproc` on the
CLI path, the OTP stderr warning, the L-0022 `RELEASE_*` env leak, the brew
`Kazi.Repo`-not-started crash) -- all four GREEN in `mix test` today. Mobile / TUI /
API-contract / Lighthouse stay documented `custom_script` recipes (ADR-0053 §3),
promoted only when a concrete goal earns it.

Existing seams reused (no new mechanism): `priv/browser/playwright_runner.js` (the
verbatim-config assertion vocabulary), `Kazi.Providers.CommandRunner` (the single
command seam), envelope-v2 score/evidence (ADR-0041), the injectable-`cmd` test seam
(so `mix test` stays hermetic), the kind->provider registry in `lib/kazi/runtime.ex`,
loader validation in `lib/kazi/goal/loader.ex` (`validate_provider_config/3`), and
`kazi schema` provider help in `lib/kazi/predicate/schema.ex` (`@schemas`) --
NOT `lib/kazi/cli/schema.ex`, which is the `--json` RESULT envelope (T16.1,
ADR-0024) and carries no provider help; T43.1 verified this seam in practice.
`:cli` mirrors `:cve` in structure.

**LANDMINE for T43.2-T43.5 (each adds an assertion type; found in T43.1).** A new
assertion type must be added in LOCKSTEP to BOTH sides or it is rejected at
goal-load: the runner's `ASSERTIONS` dispatch table in
`priv/browser/playwright_runner.js`, AND `@browser_assertion_types` in
`lib/kazi/goal/loader.ex`. Adding only the runner entry makes the type work in
the runner but fail validation the moment a goal-file names it. The lockstep is
documented in both files and pinned by a test -- know it before you start, not
after.

**Waves.** Wave A (UI pack) and Wave B (`:cli`) are independent and run in parallel.
Within Wave A, T43.1 lands the runner assertion-dispatch extension point + the first
type, and T43.2-T43.5 build on it (they touch the same runner/schema/loader files, so
they serialize behind T43.1 to avoid churn); T43.6 closes the wave with docs + a live
dogfood. Within Wave B, T43.7 is the provider skeleton and T43.8/T43.9 depend on it.
Wave C (T43.10, deferred-surface recipes) is standalone and can run any time.
