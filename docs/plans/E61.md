# E61 -- Ship kazi as a Claude Code plugin: skill + MCP + hooks in one installable, auto-updating artifact (P2)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

Issue #1173 (filed by the maintainer, 2026-07-xx, kazi 1.160.0): onboarding a
Claude Code user to kazi today takes THREE separate opt-in commands
(`install-skill`, `init --with-mcp` per repo, `install-hooks`), and the
teaching surfaces only refresh when the user remembers to re-run them after
a binary upgrade. This is a recurring, observed staleness class -- #956 (a
tiering fix stranded in a local dotfile) and #1166 (E56/T56.2, the embedded
skill teaching workarounds for flags the binary now has). Claude Code
plugins are the platform's native answer: one install, marketplace-driven
updates, and (per the issue) official plugins already bundle MCP servers +
skills and even manage server binaries, so the pattern is established
platform-side, not a novel integration.

The proposal (full detail in issue #1173's body -- read it before starting,
this epic's tasks summarize but do not replace it): a `kazi` Claude Code
plugin GENERATED BY CI from the same sources the existing installers use
(`Kazi.Teach.InstallSkill`'s rendered SKILL.md/AUTHORING.md/RECIPES.md, the
`kazi mcp` stdio registration `init --with-mcp` already writes, the bus
hooks T55.9 already implements), published to a `kazi-org` marketplace repo
on every release. The plugin version IS the release version -- the release
pipeline regenerates and publishes it, so marketplace content can never lag
the binary. Existing installers stay as a parallel channel for
Codex/Cursor/other harnesses and users who prefer explicit commands; this is
an additional channel rendered from the same single source of truth, never
a fork of it. Consent-first (ADR-0024) is preserved: installing the plugin
IS the opt-in.

The ADR is docs/adr/0077-claude-code-plugin-distribution.md (T61.1): a new
distribution channel with its own versioning/lockstep contract, not covered
by any existing ADR.

Acceptance: a `kazi` plugin manifest exists and validates against the Claude
Code plugin schema; installing it bundles the rendered skill, the MCP server
registration, and the bus hooks in one operation; the plugin version is
generated and published automatically by the release pipeline (never
hand-maintained); a SessionStart check detects binary/plugin version skew
and emits a one-line warning; `LOCAL.md`'s path is migrated per the ADR-0074
amendment so a plugin update (which replaces skill dirs wholesale) never
destroys operator customization; existing installers are unchanged and still
work standalone.
