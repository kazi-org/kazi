# E53 -- Reliability hardening: the four open runtime bugs from the E50 execution sweep (P0/P1)

Back-link: [docs/plan.md](../plan.md) (master index). Sibling epics: [plans/](.)

fidelity: executable

The 2026-07-09/10 E50 execution sweep (7 goal-files driven to merge) surfaced
four runtime bugs, triaged 2026-07-10. #1027 is a correctness gap in what
`converged` means (a second reporter saw `landed` silently PASS on a
substituted `kazi/integrate-*` branch while the named task branch had zero
commits); #1022 destroyed a live run's worktree mid-iteration; #1020 stalls
runs indefinitely at the integrate boundary; #1013 reds roughly half of all
required-check CI runs, which normalizes admin-merge bypasses (it blocked a
docs-only PR 3/3 on 2026-07-10). Every future fleet run hits some of these.

Acceptance: each fix ExUnit-pinned; integrate is idempotent for already-
landed work; a live run's worktree is never reaped; an integrate stall
self-diagnoses with a bounded timeout + captured stacktrace instead of
hanging; the session-id capture path is synchronous-or-drained with
diagnostic logging so the CI flake either disappears or names its cause.

- [ ] T53.1 Integrate idempotence -- already-landed work mints no new branch (#1027): in `Kazi.Actions.Integrate` (lib/kazi/actions/integrate.ex; the default branch mint `kazi/integrate-<ts>` is ~176), detect "already landed" BEFORE any branch creation/rename: working tree clean AND the current branch has an upstream AND HEAD == `@{u}` -> integrate returns success as a NO-OP (idempotent), never minting a fresh integrate branch; likewise a workspace already on a pushed-and-current `task/*` branch is left exactly where it is. Preserve T50.2's `kazi-partition/` discriminator in `Kazi.Scheduler.SerialLanding` untouched (its tests are a guard). Update the moduledoc contract. Owner: pool Est: 3h kind: agent verifies: [UC-036] deps: [] acc: ExUnit (fixture remote) -- first integrate on dirty work behaves as today (branch, commit, push); a second integrate with clean tree + HEAD==@{u} creates NO new ref (for-each-ref count pinned) and reports success; a pushed-and-current task/* workspace is untouched; SerialLanding + integrate suites stay green.
- [ ] T53.2 Worktree liveness guard -- never reap a live run's worktree; a vanished workspace is a distinct fatal cause (#1022): (a) every removal path in `Kazi.Scheduler.Worktree` (`reap/2` ~184 and the terminal-state cleanup) checks run liveness (the WorktreeTable row's owning run/partition state) and SKIPS + logs live entries instead of removing; (b) the loop detects "workspace disappeared" (workspace dir absent, or git exit 128 with a not-a-repository/deleted-cwd signature) and terminates with a distinct stuck cause (`:workspace_missing`, ADR-0058 permanence: permanent) plus a one-line remedy (the .git/worktrees admin-data restore from issue #1022), instead of grinding predicate iterations against a dead path. Owner: pool Est: 3h kind: agent verifies: [UC-036] deps: [] acc: ExUnit -- reap skips a worktree whose owning run is live (and reaps it once terminal); a run whose workspace dir is deleted between iterations terminates with the distinct cause and remedy in the result, with no further dispatch.
- [ ] T53.3 Integrate stall self-diagnosis -- bounded timeout + captured stacktrace (#1020): wrap the `:integrate` action execution (lib/kazi/loop.ex ~1048, `handle_event(:internal, {:act, %Action{kind: :integrate}}, ...)`) in a bounded timeout (config `integrate_timeout_ms`, default 15 min); on expiry capture the acting process's `:current_stacktrace` + message-queue length into the run's evidence/sink, kill the action task, and terminate with stuck cause `:integrate_timeout` (transient per ADR-0058 -- kill+relaunch is the known-good remedy) instead of hanging forever at 0% CPU. Normal integrates are unaffected. Owner: pool Est: 2.5h kind: agent verifies: [UC-036] deps: [T53.1] acc: ExUnit -- a stub integrate action sleeping past a short test timeout yields `:integrate_timeout` with a stacktrace in evidence and the loop terminal (not hung); a fast integrate is unchanged; the timeout is configurable.
- [ ] T53.4 Session-id capture: synchronous-or-drained + diagnostic logging (#1013): make `harness_session_id` recording robust on the terminal path -- the dispatch report's session_id is projected synchronously before the run's terminal projection (or finalize drains pending dispatch reports first), and every silent drop point on that path (envelope parse yielding no session_id, a best-effort registry write swallowing an error) logs a WARN naming what was dropped; harden test/kazi/integration/run_registry_wiring_test.exs to await the projection rather than reading immediately. The issue's async-race theory was DISPROVEN (see #1013 comments) -- so the deliverable is belt (ordering guarantee), suspenders (the awaits), and instrumentation (the WARNs) such that if Linux CI still reds, the failure log names the actual drop point.
